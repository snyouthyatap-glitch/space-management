## 현재 상태

QR 웹사이트의 화면과 주요 기능 코드는 대부분 구현되어 있습니다.

다만 실제 운영 기준으로 보면 아래 두 가지가 아직 핵심 이슈입니다.

1. 브라우저에서 Firestore를 직접 읽고 쓰는 구조
2. 관리자 패스워드 확인이 클라이언트 쪽에서 이뤄지는 구조

## 왜 이게 중요한가

현재 브라우저는 아래 작업을 직접 처리하려고 합니다.

- 전화번호 기준 회원 조회
- 라운지 임시회원 생성
- 방문일지 / 시설이용일지 저장

이 구조를 로그인 없이 동작시키려면 Firestore 권한을 꽤 많이 열어야 합니다.

그렇게 되면 다음 문제가 생길 수 있습니다.

- 외부에서 회원 조회를 시도할 수 있음
- 가짜 방문 기록이나 시설 이용 기록을 반복해서 넣을 수 있음
- 관리자 패스워드를 프론트 또는 Firestore에서 비교하면 진짜 비밀값으로 보호되기 어려움

## 운영용으로 추천하는 방향

공개 QR 흐름은 브라우저가 Firestore를 직접 만지지 않도록 바꾸는 것이 좋습니다.

중간에 서버 역할을 하는 계층을 하나 두는 방식을 추천합니다.

예시:

- Google Apps Script Web App
- Firebase Cloud Functions
- 기타 간단한 백엔드

브라우저는 이 중간 계층에 요청만 보내고,
실제 Firestore 읽기/쓰기와 관리자 검증은 서버 쪽에서 처리하는 방식이 안전합니다.

## 서버 쪽으로 옮기면 좋은 작업

다음 작업들은 브라우저가 직접 하지 말고 서버 쪽에서 처리하는 것이 좋습니다.

1. 시설 회원 전화번호 / 끝자리 조회
2. 시설 회원 자동 가입
3. 라운지 임시회원 생성
4. 방문일지 생성
5. 관리자 패스워드 검증

이렇게 바꾸면 브라우저는 단순히 입력값만 보내고,
실제 데이터 접근은 신뢰 가능한 쪽에서 처리하게 됩니다.

## 지금 바로 테스트만 하고 싶다면

이 저장소에는 `firestore.rules` 초안이 들어 있습니다.

하지만 이 규칙은 어디까지나 테스트 출발점일 뿐이고,
실제 운영 환경에 그대로 쓰기에는 안전하지 않습니다.

## 기존 회원 데이터 정리 방향

기존 `/users/{uid}` 문서는 새로 만들기보다 기존 문서를 정리해서 재사용하는 것이 좋습니다.

권장 정리 방식:

- `phone` => 숫자만 남기기
- `phoneLastDigits` => 전화번호 끝 4자리
- `role` => `"user"`
- `status` => `"approved"`

즉, 기존 회원은 문서 ID를 유지한 채 필요한 필드만 보정하면 됩니다.
